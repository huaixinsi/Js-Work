# DeepSeek 电影助手设计

## 目标

为已登录的普通用户增加一个电影助手。助手使用 DeepSeek API，但只基于项目 MySQL 中的电影、评分、简介和本地评论样本回答最近热播、影片风评和个性化推荐问题。

## 用户体验

- 普通用户导航新增“电影助手”页，管理员端不显示。
- 页面包含对话记录、输入框、发送按钮和三个快捷问题。
- 快捷问题覆盖“最近有什么热门电影”“某部电影风评如何”“根据观影想法推荐电影”。
- 用户发送问题后显示等待状态；请求期间禁止重复发送。
- 回答明确说明口碑数据来自本项目的本地评论样本。
- 错误信息显示在对话区域，不暴露服务端密钥或上游响应细节。

## 后端接口

新增 `POST /api/assistant/chat`：

- 必须携带普通用户 JWT，管理员和匿名请求不能调用。
- 请求体包含最近对话消息，最多 8 条。
- 消息角色仅允许 `user` 和 `assistant`，每条内容为 1 至 500 个字符。
- 返回 `{ answer: string }`。
- 请求不在数据库持久化，刷新页面后会话清空。

## 本地电影上下文

每次请求从 MySQL 读取最多 50 部可展示电影，按热度和评分排序。上下文包括：

- 电影 ID、标题、上映年份、类型标签。
- TMDB 评分、热度、简介。
- 本地评论数量、平均分和最多两条评论摘要。
- 评论摘要标记为“本地演示样本”，不能描述为全网真实口碑。

系统提示要求模型：

- 只回答电影、观影和购票相关问题。
- 优先引用上下文中真实存在的电影，不编造片名、评分或评论。
- 推荐时说明简短理由，并优先给出 3 至 5 部电影。
- 当本地数据不足时明确说明，不假装掌握实时互联网信息。

## DeepSeek 集成

- 服务端直接使用原生 `fetch` 调用 `https://api.deepseek.com/chat/completions`，不新增 SDK。
- 默认模型为 `deepseek-v4-flash`，可通过 `DEEPSEEK_MODEL` 覆盖。
- 密钥读取 `DEEPSEEK_API_KEY`，仅存在于本地 `.env` 和容器环境。
- 请求使用非流式响应，降低前端和 Express 接口复杂度。
- 携带系统提示、本地电影上下文和最近 8 条对话消息。
- 使用 30 秒超时；上游非成功响应、空回答或格式异常统一转换为用户可理解的错误。

## 配置与安全

- `.env` 写入真实 `DEEPSEEK_API_KEY`，该文件已被 Git 和 Docker 构建上下文忽略。
- `.env.example` 只加入示例配置和默认模型名，不包含真实密钥。
- `docker-compose.yml` 将 `DEEPSEEK_API_KEY` 和 `DEEPSEEK_MODEL` 注入应用容器。
- 前端永远不读取或返回 DeepSeek 密钥。
- 接口通过现有 JWT 身份验证和角色校验保护。
- 限制消息数量和长度，避免异常请求造成不必要的 API 消耗。

## 模块边界

`server/domain/assistant.js`：

- 校验并规范化聊天消息。
- 将数据库电影及评论聚合结果格式化为紧凑上下文。
- 构造系统提示。

`server/domain/deepseek.js`：

- 构造 DeepSeek Chat Completions 请求。
- 管理超时、响应解析和错误归一化。
- 支持注入 `fetch`，便于单元测试。

`server/routes.ts`：

- 执行 JWT 和用户角色校验。
- 查询本地电影与评论聚合。
- 调用领域模块并返回回答。

`src/App.tsx`、`src/api.ts`、`src/styles.css`：

- 增加电影助手导航、状态、聊天界面和 API 调用。
- 沿用现有视觉体系，不重做其他页面。

## 测试

- 消息角色、数量、空内容和 500 字上限校验。
- 电影上下文包含标题、评分、评论摘要和演示样本标识。
- DeepSeek 请求使用正确 URL、模型、认证头和消息格式。
- 上游超时、错误状态和空回答会返回稳定错误。
- 运行全部单元测试和 TypeScript/Vite 构建。
- Docker 重建后使用普通用户 JWT 发起真实电影问题，验证回答非空。
- 验证管理员和匿名请求无法调用助手接口。
- 验证 `.env` 未被 Git 跟踪，Git diff 中不包含真实密钥。

## 验收标准

- 普通用户登录后可进入“电影助手”并连续提问。
- 助手能根据本地电影数据回答热门、风评和推荐问题。
- 回答不会把演示评论冒充为全网真实口碑。
- DeepSeek 密钥不出现在前端、Git 历史、日志或 API 响应中。
- 单元测试、生产构建、Docker 和真实 API 联调全部通过。
